# Stash 懒猫微服移植计划

本文档参考 `开源项目移植到懒猫微服通用指南-3.5.md`，并按懒猫微服 LPK V2 结构规划：

- `package.yml`：应用元数据和权限声明
- `lzc-manifest.yml`：运行时服务、路由、认证、注入配置
- `lzc-build.yml`：正式打包配置
- `lzc-build.dev.yml`：开发态部署覆盖配置

## 目标

把当前 Stash 多容器应用打包成懒猫微服 LPK，并满足以下验收条件：

- 用户通过懒猫应用入口访问 Stash，不需要再手动输入 Stash 账号密码。
- Web UI、后端 API、协作 WebSocket 都走同一个懒猫应用域名。
- Postgres、Redis、后端密钥文件都能持久化。
- `lzc-cli project deploy` 可在开发态部署验证。
- `lzc-cli project release` 可产出正式 LPK 包。

## 当前项目结构判断

当前生产部署在 `docker-compose.prod.yml` 中定义了这些服务：

| 服务 | 作用 | 端口 | 移植处理 |
| --- | --- | --- | --- |
| `postgres` | 主数据库，使用 pgvector | 5432 | 保留为内部服务，持久化 `/var/lib/postgresql/data` |
| `redis` | 队列和缓存 | 6379 | 保留为内部服务，持久化 `/data` |
| `backend` | FastAPI API 和迁移入口 | 3456 | 保留为内部服务 |
| `worker` | Celery worker | 无 HTTP | 保留为内部服务 |
| `beat` | Celery beat | 无 HTTP | 保留为内部服务 |
| `frontend` | Next.js 产品 UI | 3457 | 作为主 Web 入口 |
| `collab` | Yjs WebSocket 协作服务 | 3458 | 通过 `/collab` 转发 |
| `caddy` | 自托管反向代理 | 80/443 | 懒猫版本中删除，由懒猫网关负责路由 |

不建议直接把 `docker-compose.prod.yml` 原样转换成 LPK。原因是 Caddy、域名、TLS 和公网端口在懒猫环境中应交给平台网关处理，否则会多维护一层入口配置。

## 关键技术方案

### 1. 路由方案

在 `lzc-manifest.yml` 中用懒猫网关替代 Caddy：

- `/api/*` 转发到 `http://backend:3456`，保留 `/api` 前缀。
- `/health` 转发到 `http://backend:3456/health`。
- `/collab*` 转发到 `http://collab:3458`，用于 WebSocket。
- `/` 转发到 `http://frontend:3457`。

涉及前缀的路由使用 `upstreams`，并设置 `disable_trim_location: true`，避免后端收到被裁剪后的路径。

### 2. 域名和环境变量

懒猫会注入 `LAZYCAT_APP_DOMAIN`。Stash 懒猫版本应统一使用：

- `PUBLIC_URL=https://${LAZYCAT_APP_DOMAIN}`
- `CORS_ORIGINS=https://${LAZYCAT_APP_DOMAIN}`
- `BACKEND_INTERNAL_URL=http://backend:3456`
- `BACKEND_URL=http://backend:3456`
- `NEXT_PUBLIC_COLLAB_URL` 留空，让前端按当前域名生成 `/collab` WebSocket 地址

OAuth 回调类变量如果启用，需要全部改成懒猫域名下的 HTTPS 回调地址。未配置对应 OAuth client 时保持为空，功能应明确不可用。

### 3. 免密登录方案

优先采用懒猫 HTTP Header 身份识别，而不是 Inject 自动填表。

懒猫网关会给请求注入：

- `SAFE_UID`
- `SAFE_USER_GROUPS`

计划新增一个明确的懒猫认证模式，例如 `LAZYCAT_AUTH_ENABLED=true`：

- 后端在该模式下从请求 Header 解析 `SAFE_UID`。
- 第一次访问时按 `SAFE_UID` 创建或更新本地用户。
- `SAFE_USER_GROUPS` 包含 `ADMIN` 时映射为 Stash 管理员角色。
- 前端在该模式下不再要求 `localStorage` 中存在 `stash_token`，直接调用 `/api/v1/users/me`。
- 原登录页在该模式下隐藏密码登录和注册入口。

这样可以满足免密登录要求，同时保持浏览器请求和现有 API 路由结构简单。

CLI 和外部 agent 仍需要 API key。懒猫登录只负责浏览器会话，用户进入设置页后再按现有流程生成 CLI key。

### 4. 持久化数据

LPK 中需要保留这些持久化目录：

- Postgres：`/var/lib/postgresql/data`
- Redis：`/data`
- Backend secrets：`/app/.secrets`

`backend_secrets` 不能丢失，否则自动生成的 `INTEGRATIONS_ENCRYPTION_KEY` 会变化，已保存的集成 token 可能无法解密。

### 5. 镜像策略

第一阶段使用当前生产镜像，降低移植变量：

- `ghcr.io/fergana-labs/stash-backend:0.1.111`
- `ghcr.io/fergana-labs/stash-frontend:0.1.111`
- `ghcr.io/fergana-labs/stash-collab:0.1.111`
- `pgvector/pgvector:pg16`
- `redis:7-alpine`

第二阶段再决定是否改为 `lzc-build.yml` 内构建并嵌入镜像。正式发布前不应使用浮动 tag。

## 文件实施计划

### 阶段 1：新增 LPK 骨架

新增这些文件，统一放在 `lazycat/` 目录：

- `lazycat/package.yml`
- `lazycat/lzc-manifest.yml`
- `lazycat/lzc-build.yml`
- `lazycat/lzc-build.dev.yml`
- `lazycat/lzc-icon.png`

`package.yml` 初始建议：

- `package: cloud.lazycat.app.stash`
- `name: Stash`
- `version` 与当前镜像版本保持一致，例如 `0.1.111`
- `permissions.required` 至少包含 `net.internet`

如果后续要写入懒猫文稿目录，再增加 `document.private`。当前 Stash 数据主要在数据库和对象存储中，不先申请文稿权限。

### 阶段 2：定制懒猫运行配置

从 `docker-compose.prod.yml` 迁移服务定义，但做这些调整：

- 删除 `caddy` 服务。
- 删除对 `./Caddyfile` 的挂载。
- 所有对外 `ports` 改为内部服务端口或懒猫路由。
- 使用懒猫域名注入 `PUBLIC_URL` 和 `CORS_ORIGINS`。
- 给 `backend`、`worker`、`beat` 注入一致的 Stash 环境变量。
- 保留 `postgres`、`redis`、`backend` 的健康检查。
- 确保 `backend` 仍在启动生命周期中执行 Alembic 迁移。

### 阶段 3：接入懒猫免密登录

后端改动范围：

- 在认证模块中增加懒猫 Header 认证分支。
- 增加按 `SAFE_UID` 查找或创建用户的服务函数。
- 给 `users/me`、workspace 初始化和邀请转换补测试。
- 缺少 `SAFE_UID` 时直接返回 401，不做静默降级。

前端改动范围：

- 增加 `NEXT_PUBLIC_LAZYCAT_AUTH_ENABLED`。
- 认证 hook 在懒猫模式下允许无 token 调用 `/users/me`。
- 登录页在懒猫模式下改为跳转主界面或展示明确错误。
- 设置页保留 API key 管理能力，供 CLI 使用。

### 阶段 4：本地和懒猫开发态验证

本地验证：

- `docker compose -f docker-compose.prod.yml up -d` 作为基线。
- `pytest` 验证后端认证和用户创建逻辑。
- `cd frontend && npm test` 验证前端认证分支。
- 用浏览器验证首页、登录态、设置页、session 页面、文件上传入口。

懒猫开发态验证：

- `lzc-cli project deploy`
- 查看所有服务状态和日志。
- 访问懒猫分配域名。
- 验证 `/api/v1/users/me` 能通过懒猫 Header 返回当前用户。
- 验证 `/collab` WebSocket 能连接。
- 重启应用后确认数据库数据和 backend secrets 未丢失。

### 阶段 5：正式打包和回归

正式打包：

- `lzc-cli project release`
- 确认产物为 LPK V2 tar 格式。
- 安装到干净懒猫环境。

回归清单：

- 首次打开应用自动进入 Stash。
- 新用户自动创建个人 workspace。
- 创建 session、上传文件、创建 page 均正常。
- 搜索和向量索引任务正常入队并执行。
- 协作编辑页面无 WebSocket 断连。
- 未配置 `ANTHROPIC_API_KEY` 时 ask-the-stash 明确不可用。
- 配置 `ANTHROPIC_API_KEY` 后 ask-the-stash 可用。
- OAuth 集成未配置 client 时不展示误导性的可连接状态。

## 风险和待确认点

### 懒猫 Header 是否会转发给所有 upstream

需要在开发态确认 `SAFE_UID` 和 `SAFE_USER_GROUPS` 是否会到达 `/api` upstream。若只到达主应用服务，需要改用 OIDC 或 request inject 转发身份。

### WebSocket 路由

`/collab` 需要确认懒猫网关对 WebSocket upgrade 的支持和路径保留行为。若路径被裁剪，必须使用 `disable_trim_location: true`。

### OAuth 回调

Google、Gmail、GitHub、Notion、Linear、Jira、Asana 等外部 OAuth 回调必须使用懒猫 HTTPS 域名。未配置时先保持不可用，不做兼容 fallback。

### 本地 embedding 模型

无 OpenAI/HF token 时后端会使用本地 sentence-transformers，首次启动可能下载模型。懒猫环境如果无法访问模型源，需要在镜像中预置模型或要求配置 hosted embedding provider。

### 镜像来源

使用 GHCR 镜像最简单，但正式上架可能需要镜像可稳定拉取或内嵌。开发验证通过后再决定是否转为 `embed:` 镜像构建。

## 推荐执行顺序

1. 先新增 LPK 骨架和懒猫路由，使用现有密码登录跑通应用。
2. 再加入 `LAZYCAT_AUTH_ENABLED` 后端 Header 认证。
3. 然后更新前端认证分支，完成免密登录。
4. 最后做开发态部署、重启持久化验证和 release 打包。

这个顺序能把问题拆开：先证明容器和路由能跑，再处理认证，最后处理发布质量。

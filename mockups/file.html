Stripe webhook handling

Every billing state change flows through the /webhooks/stripe endpoint. We verify the signature, then dispatch on event.type. A 400 (bad signature) response is healthy — it means an unsigned probe hit the endpoint.

STRIPE_SECRET_KEY is the on/off switch. With no key set, the connect-time 402 gate is bypassed and billing is effectively off.

The 402 connect gate

Per-user Pro is enforced at connect time. The flow is upsert-aware, so reconnecting an existing user never double-charges:

# verify, then gate
if stripe_enabled() and not user.is_pro:
    raise HTTPException(402, "upgrade required")

Renewal failures

  • Invoice payment_failed → retry schedule kicks in (3 attempts over 7 days)
  • After final failure → workspace flips to read-only, owner is emailed
  • Manual recovery → re-run the connect flow once the card is updated
